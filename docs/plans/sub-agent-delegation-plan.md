# 子 Agent 分派机制 - 计划文档

> 状态: 归档中，暂未实现

---

## 1. 背景与问题

当 AI 调用工具时，某些工具可能会返回**大量数据**（如搜索整个代码库、列出所有文件、获取完整日志等）。这些数据会快速填满上下文窗口，导致：

- 后续对话无法继续
- 重要信息被淹没在噪声中
- AI 无法有效利用返回的数据

---

## 2. 解决方案：通用的子 agent 分派框架

**设计思路**：主 agent 不应该"自己一个人干完所有事"，而是把任务**分派**给合适的子 agent。主 agent 负责**调度**，子 agent 负责**执行**。

这不是"数据提取"这一件事，而是**任务分派**的通用能力：

- 数据量太大？→ 分派给子 agent 提取关键信息
- 需要深度研究？→ 分派给子 agent 深入分析
- 需要并行探索？→ 分派多个子 agent 同时搜索
- 需要专家咨询？→ 分派给专家型子 agent

---

## 3. 子 Agent 类型说明

| 子 agent      | 用途          | 适用场景                         |
| ------------- | ------------- | -------------------------------- |
| `refine_data` | 数据提取/过滤 | 工具返回大量数据，只取需要的部分 |
| `explore`     | 代码库探索    | 搜索模式、查找实现、理解结构     |
| `librarian`   | 外部知识      | 查文档、搜 GitHub、找最佳实践    |
| `deep`        | 深度问题解决  | 复杂逻辑、架构决策、疑难 bug     |
| `parallel_*`  | 并行任务      | 多个独立任务同时执行             |

---

## 4. 通用的子 agent 调用模式

```
┌─────────────────────────────────────────────────────────────┐
│                     主 Agent (Master)                         │
│  - 理解任务                                                 │
│  - 分解任务                                                 │
│  - 分派给合适的子 agent                                     │
│  - 收集结果，整合答案                                        │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  子 Agent 类型 (按需选择)                                    │
│  ─────────────────────────────────────────────────────────  │
│  • refine_data   → 数据提取/过滤                            │
│  • explore       → 代码库探索/搜索                          │
│  • librarian     → 文档查询/外部知识                        │
│  • deep          → 深度问题解决                            │
│  • parallel_*    → 并行执行多个任务                        │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  返回精炼结果给主 agent                                      │
└─────────────────────────────────────────────────────────────┘
```

---

## 5. 何时应该分派子 agent

### 主 agent 应该主动分派 when:

- 任务可以分解为独立子任务
- 需要搜索/探索多处代码
- 工具返回数据量过大
- 需要专家知识（查文档、找最佳实践）
- 任务需要并行加速

### 不要分派 when:

- 任务非常简单明确
- 已有足够上下文可以直接回答
- 需要主 agent 亲自判断的场景

---

## 6. 工具的 refine_required 标记

在工具定义处添加 `refine_required: true` 标记，提醒 AI 此工具可能返回大量数据，需要考虑分派子 agent：

```json
{
  "name": "grep",
  "description": "搜索代码库中的文本模式...",
  "refine_required": true,
  "recommendation": "此工具可能返回大量匹配结果，考虑分派子 agent 提取需要的文件路径"
}
```

## 7. 实际使用示例

### 示例 1: 代码搜索 → 分派子 agent 提取

```
用户: 找出所有使用 useState 的 React 组件

1. 主 agent: 任务可以分解，先搜索，再提取
2. 调用 grep → 返回 500+ 行结果
3. 判断: 数据量过大，需要子 agent 处理
4. 分派 refine_data(goal="提取文件路径列表")
5. 使用精炼结果继续工作
```

### 示例 2: 复杂需求 → 并行分派

```
用户: 比较 React 和 Vue 的状态管理方案

1. 主 agent: 这是研究任务，可以并行
2. 分派 librarian(react 状态管理)
3. 分派 librarian(vue 状态管理)
4. 收集两个结果，整合比较
```

### 示例 3: 多处探索 → 并行分派

```
用户: 理解这个项目的认证流程

1. 主 agent: 需要了解多处代码
2. 分派 explore(认证中间件)
3. 分派 explore(登录/登出逻辑)
4. 分派 explore(Token 生成/验证)
5. 收集结果，理解全貌
```

## 8. 最佳实践

1. **主动分派**: 不是等数据来了再处理，而是预判是否需要分派
2. **选择合适的子 agent**: 根据任务性质选择最专业的
3. **保持主 agent 的责任感**: 主 agent 对最终答案负责，不是甩手掌柜
4. **迭代 refinement**: 第一次结果不满意，可以再次分派子 agent 优化

```
主 agent 的角色: 指挥官 + 整合者
子 agent 的角色: 专业执行者
```
